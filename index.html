I created the Synthetic Data set to test my classification algorithm and this visualization. See how:

  • The first five sets (set1 - set5) are each composed of 50 random IP addresses, all with the subnet '/32.' It is extremely unlikely for any set to be equal to or include one of these sets. These sets are later used as a control to generate future sets and test whether the visualization works properly.
  • The next twenty sets are created by taking each of the first five sets, and adding four sets for each in which the elements' subnets are changed to '/24, /16, /8, /0.' These sets are used to check intersection and inclusion. If you select the Synthetic Data and group and sort by 'name', notice the dark green vertical lines on the right side of the matrix, where the sets on the x-axis include those in the y-axis (set9, set13, set17...). These lines are the sets whose subnets were changed to '/0', which is why they include those on the y-axis.
  • Sets 26 through 30 are identical to the first five, except for that the elements are shuffled within the set. This is to test whether my algorithm recognizes a set { a, b, c } as equal to { b, c, a }, which it should. If you view the synthetic data sorted by name, you will see that the there is a diagonal of five black cells indicating 'set1' = 'set26', 'set2' = 'set27', and so on. It is (very) nearly impossible that this is a cooincidence, because the first five sets are composed of 50 random 32-bit addresses, each with subnet 32.
  • Sets 31 through 35 are identical to sets 1 through 5, with one aditional random element per set. These sets are used to test inclusion. If you view the synthetic data sorted by name, notice the diagonal line of five light brown boxes in th top right of the matrix, indicating 'set31' includes 'set1,' 'set32' includes 'set2,' and so on. Again, it's nearly impossible that this is cooincidence.
  • Sets 41 through 45 are created by taking each of the first five sets, and removing one element from each. This is to check inclusion in the other direction. Notice that 'set1' includes 'set41', 'set2' includes 'set42', and so on.
  • The final five sets are identical to the sets 41 through 45, except that one element is randomly deleted and another is randomly added to each set. This is to test intersection once again. Notice that 'set1' and 'set41' intersect, and so on.

This visualization, called an adjacency matrix, has certain advantages over visualizations like venn diagrams in showing relationships between sets of objects. For one, you can see how two distinct sets are related, and how sets are related to those near them. Displaying the data in a symmetric matrix provides the user certain intuition about the data. For example, if you view the Synthetic Data and sort by 'name,' the obvious symmetry and linearity of the data indicates that it is not random an that there is an underlying pattern. Also, the area of the matrix covered by a color, which is immediatly recognizable, corresponds to the frequency at which that group appears. If you view the Symmetric Data and group by 'cluster,' it's clear from the area of different shades that this data set is split nearly pefectly in four. Though venn diagrams are effective with very small data, they quickly lose their value with larger data. How would you show the relationships between fifty sets (as the above matrix does) with a venn diagram?

There are also certain disadvantages in using an adjacency matrix. The clustering feature makes it so there will almost always be a level diagonal symmetry that will catch the eye, regardless of the data. This may mislead a user into thinking that there is some underlying relationship in the data when there isn't. A good example is the Random Data set when sorted by 'cluster' – if unfamiliar with the data beforehand, one might think it is highly structured and normal after seeing its symmetry.

In terms of run-time, here is how my algorithm performed. I tested on variously sized collections of sets, each set consisting of 50 addresses. Because some checks are more time consuming than others depending on the subnet (for example, checking intersection is harder when the subnets are high, while checking equality is harder when they are low), the values of the subnets has an impact on run time. I tested at varying subnets as well. I am running a 2013 MacBook Pro, 2.4 GHz dual core processor, 8 GB 1600 MHz memory.
